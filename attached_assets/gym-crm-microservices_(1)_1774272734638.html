      How data flows between actors and services in real-time operations.
      👑 GYM OWNER
      Login → Auth Service → HQ Dashboard (Tenant: ALL) → HQ Analytics aggregates all branches + franchises → Franchise Mgmt: set packages, royalty rules → Reporting: consolidated P&L, turnover
      🏢 BRANCH MANAGER
      Login → Auth (role: branch_manager, tenant: branch_X) → Member Service: enroll/manage members → Billing: collect payments → Event Bus: payment_completed event → HQ Analytics receives aggregated data
      🤝 FRANCHISEE
      Login → Auth (role: franchise_manager, tenant: franchise_Y) → Packages: ONLY from approved HQ catalogue → Billing: payments → royalty split auto-calculated → Event Bus: revenue_reported event → Owner sees ALL franchise turnover in HQ Analytics
      👤 GYM MEMBER
      App Login → Auth → View own package & schedule → Attend class → Attendance logged → Billing: autopay subscription → Diet & workout plans via trainer
      KEY DESIGN PRINCIPLE: Multi-Tenancy + Owner Override
      Every service request carries a tenant_id in the JWT token. Branch managers and franchise managers only see data for their own tenant_id. The GYM owner carries a special role: owner + tenant_id: * (wildcard), which tells every service to return data across all tenants. The HQ Analytics and Reporting services then aggregate and visualize this holistically.
      Each microservice owns its database. No shared DB access between services — data flows via API calls or events only.
